Linux-first workflow for Vision Pro workspace development

Vision Web Workspace

Browser-simulated spatial web desktop with governed handoff to a real Mac Builder.

Vision Web Workspace is a development foundation for Apple Vision Pro style spatial web workspaces. It defines workspace, window, session, and Mac Builder contracts on Linux, validates movable web windows and remote development surfaces in a browser simulator, and hands native visionOS build, simulator, archive, signing, artifact, and `.xcresult` work to a controlled remote Mac execution plane.

Simulate before Mac

Validate spatial workspace interaction before native build access exists.

The Linux loop covers move, resize, focus, layout persistence, terminal windows, code windows, and remote browser sessions.

Web windows as workspace

Each window is a controlled browser-addressable development surface.

Windows can host SSH terminals, code-server, documentation, CI dashboards, logs, notebooks, or remote browser streams.

Governed Mac handoff

Native Apple work moves through an explicit Mac Builder protocol.

Linux submits build, test, and archive requests; macOS workers own XcodeGen, xcodebuild, simulator, artifacts, and `.xcresult` output.

Problem Architecture Workflow

Problem

Without a Mac, visionOS development cannot start from native compilation.

Official visionOS compilation, signing, simulator execution, and device testing require macOS, Xcode, the visionOS SDK, signing assets, and Apple account setup. Product interaction, window models, remote development surfaces, and build request protocols can still be designed and validated before that native toolchain is available.

This repository moves the Linux-validatable work forward first: window state, web workspace interaction, session gateway, workflow checks, and Mac Builder request handling. Native build and signing remain owned by real Mac infrastructure.

Product risk it reduces

Before committing budget to EC2 Mac, local Mac hardware, signing, or device labs, the project can validate the workspace interaction model and remote development workflow.

What it is

A development platform that separates spatial web workspace design from Apple-required native execution.

The system splits product interaction, session gateway, and native build handoff into explicit layers. Linux can drive the simulator and contract loop; Mac hardware is reserved for the Apple toolchain responsibilities.

Workspace contracts

Window and session model

Shared contracts define workspace, panel, layout, session, and Mac build requests instead of hiding state inside UI components.

Browser simulator

Spatial workspace in a normal browser

The simulator validates head-locked desktop behavior, movable windows, and terminal, code, browser, docs, and logs panels.

Session gateway

Remote development session entrypoint

The gateway skeleton centralizes terminal, code, and browser session creation for later ttyd, code-server, VNC, or WebRTC integration.

Mac Builder handoff

Native build control chain

Mock and agent packages validate job submission, status polling, logs, artifacts, and `.xcresult` references.

What it is not

The boundary is part of the workflow.

This repository does not bypass Apple tooling or claim browser simulation is equivalent to a Vision Pro device.

Not a replacement for Xcode, the visionOS SDK, Apple simulator, or signing systems.
Not a substitute for enterprise-only APIs or system-level follow-window entitlements.
Not a production remote operations platform; terminals, browsers, and code sessions still require a controlled gateway.
Not App Store release authority; upload, signing, TestFlight, and review remain later Apple workflows.
Does not put SSH keys, GitHub tokens, signing assets, or production access inside the Vision Pro client.
Does not claim Linux can compile, install, or run the native visionOS app.

Operator workflow

Local development and Mac Builder validation

The local loop validates contracts, simulator behavior, and gateway surfaces. Native build checks can target a mock builder or a real Mac Builder when one exists.

  1. Install dependencies: `pnpm install`.
  2. Start the web simulator: `pnpm dev`.
  3. Run workflow checks: `pnpm workflow:check`.
  4. Start the mock builder: `pnpm dev:mac-builder:mock`.
  5. Submit a native build check: `VISIONOS_MAC_BUILDER_URL=http://127.0.0.1:3101 pnpm visionos:mac-build:check`.
